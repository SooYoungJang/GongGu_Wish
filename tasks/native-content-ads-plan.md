# Plan: 릴스·제품 상세 네이티브 미디어 광고

## 1. 계약과 설정

1. 플랫폼별 앱 ID 및 `home`, `reels`, `detail` 광고 단위 계약을 테스트로 정의한다.
2. `adsRemoved`가 모든 광고 설정보다 우선하도록 런타임 계약을 확장한다.
3. Expo config에 Android/iOS 앱 ID를 안전하게 주입하고 Production 검증을 강화한다.

검증: 설정 단위 테스트와 Preview/Production Expo config 출력.

## 2. 공통 네이티브 광고 뷰

1. Android 전용 로더를 Android/iOS 공통 네이티브 로더로 확장한다.
2. `placement`에 따라 광고 단위를 선택하는 공통 컴포넌트를 만든다.
3. 카드형 상세 레이아웃과 전체 화면 릴스 레이아웃을 제공한다.
4. 언마운트·권한 변경·타임아웃 시 광고 객체를 정리한다.

검증: 컴포넌트 테스트, 타입 검사.

## 3. 화면 통합

1. 순수 함수로 릴스 유기 콘텐츠와 광고 슬롯을 조합한다.
2. PagerView가 공구와 광고를 구분해 렌더링하고 공구 인덱스 상태는 기존 의미를 유지하도록 한다.
3. 상세 하단에 광고 로드 성공 시에만 카드가 나타나게 한다.

검증: 배치 함수 테스트, 릴스 전환 회귀 테스트, 렌더 테스트.

## 4. 품질 게이트

1. 관련 테스트, 전체 모바일 테스트, 타입 검사, 린트를 실행한다.
2. Android/iOS 공개 Expo config를 검사한다.
3. 가능한 개발 빌드에서 테스트 광고와 실패 fallback을 확인한다.
4. 변경 및 검증 결과를 GongGu Wish 위키에 기록한다.
5. 커밋·푸시 후 `develop` 대상 PR의 필수 CI를 확인하고 통과 시 머지한다.

## Risks

- 릴스 PagerView 인덱스가 광고 슬롯 때문에 기존 공구 인덱스와 달라질 수 있음: 공구 인덱스를 별도로 보존한다.
- 영상 광고가 여러 장 선로드되면 메모리가 증가할 수 있음: 현재/인접 광고 슬롯만 마운트하고 객체를 즉시 정리한다.
- iOS 앱 ID 누락 시 시작 실패 가능: 테스트 앱 ID를 안전 기본값으로 사용하고 Production은 fail-closed 검증한다.
- 구독 상태 로딩 중 광고가 잠깐 보일 수 있음: 향후 구독 시스템은 확인 전 `adsRemoved` 또는 entitlement loading gate를 전달하도록 계약을 문서화한다.
